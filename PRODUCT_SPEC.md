## 0. 핵심 결론

이 제품은 “6모 분석기”가 아닙니다.
최종 제품은 다음 구조입니다.

> **6모부터 9모, 수능까지 사용자의 성적 변화와 목표 대학 변화를 누적 관리하고, 최종 수능 이후 가/나/다군 원서 조합까지 연결하는 AI 정시 전략 플랫폼.**

따라서 6모 버전은 독립 서비스가 아니라 **수능 메인 서비스를 위한 선공개 버전**입니다.

그리고 이 모든 것을 **앱스토어 설치 없이 링크 한 번으로 진입하는 모바일 웹앱(PWA)** 으로 전달합니다. 정시 예측은 한 해에 몇 번만 쓰는 계절 상품이고, 유입의 핵심이 SNS·커뮤니티 링크 공유와 SEO이기 때문입니다. 네이티브 앱 설치 마찰은 초기 바이럴 퍼널을 무너뜨립니다.

서비스의 시즌 흐름은 이렇게 정의합니다.

```text
6모 선공개
→ 현재 위치 진단 / 유저 계정 확보 / 목표 대학 저장 / 재방문 예약

9모 베타
→ 6모 대비 변화 분석 / 수능 목표 점수 설정 / 유료화 실험

수능 메인
→ 최종 환산점수 분석 / 진학사·고속·텔레그노시스 교차검증 / 가·나·다군 조합 / 유료 리포트 판매 / 실제 합불 데이터 회수
```

이 명세의 핵심은 두 가지입니다.

> **(1) 데이터 모델과 UX를 “성적 분석 1회”가 아니라 “입시 사이클 전체” 중심으로 설계한다.**
> **(2) 전달 채널은 PWA 우선으로 설계하고, 재방문(9모·수능)은 카카오 알림톡·이메일·웹푸시로 끌어온다.**

# 1. 제품 정의

## 1.1 제품 한 문장 정의

**Pacer는 6모·9모·수능 성적을 누적 입력하면, 사용자의 점수 구조, 목표 대학 접근도, 유리한 반영비, 지원 후보군 변화를 추적하고, 수능 이후 최종 가/나/다군 원서 전략까지 AI 리포트로 정리해주는 정시 의사결정 플랫폼이다.**

## 1.2 제품이 해결하는 핵심 문제

정시 시장의 본질적 문제는 “계산기가 없다”가 아닙니다. 이미 환산점수 계산기, 입결표, 배치표, 진학사, 고속성장분석기, 텔레그노시스 같은 대안이 있습니다.

진짜 문제는 다음입니다.

1. **정보가 흩어져 있다.**
   성적표, 대학별 모집요강, 전년도 입결, 진학사 칸수, 고속성장 결과, 커뮤니티 후기, 학원 상담이 전부 따로 존재합니다.

2. **사용자가 결과를 해석하지 못한다.**
   “5칸이면 안정인가?”, “고속은 초록인데 진학사는 4칸이면 뭘 믿어야 하나?”, “영어 2등급 감점이 얼마나 큰가?” 같은 질문에 답하기 어렵습니다.

3. **6모·9모·수능이 연결되지 않는다.**
   대부분의 서비스는 특정 시점 점수만 보여줍니다. 사용자는 6모 이후 어떤 과목이 좋아졌는지, 목표 대학까지 얼마나 가까워졌는지, 수능 이후 실제 원서 조합이 어떻게 변해야 하는지 추적하지 못합니다.

4. **학부모와 학생의 언어가 다르다.**
   학생은 점수와 대학 리스트를 보고 싶어 하고, 학부모는 “지금 괜찮은 건지”, “정시를 밀어도 되는지”, “무엇을 해야 하는지”를 알고 싶어 합니다.

5. **최종 원서접수 시점의 불안이 가장 크다.**
   수능 직후에는 사용자가 이미 진학사·고속성장·텔레그노시스를 같이 봅니다. 이때 필요한 것은 또 다른 단일 예측값이 아니라, 여러 신호를 종합해주는 교차검증 리포트입니다.

## 1.3 제품의 핵심 가치 제안

### 학생에게

> “내 점수가 어느 대학에서 유리한지, 6모에서 9모, 수능까지 내 위치가 어떻게 바뀌었는지 한눈에 볼 수 있다.”

### 학부모에게

> “복잡한 입시 용어 없이, 현재 성적의 의미와 최종 원서 전략을 이해할 수 있다.”

### N수생에게

> “진학사, 고속성장, 텔레그노시스 결과가 엇갈릴 때 왜 그런지 교차검증할 수 있다.”

### 컨설턴트에게

> “학생별 성적·목표·후보군·리스크를 자동 정리해 상담 준비 시간을 줄일 수 있다.”

## 1.4 플랫폼 전략: 모바일 웹앱(PWA) 우선

Pacer는 **앱스토어 등록 없이 링크로 즉시 진입하는 모바일 최적화 반응형 웹앱(PWA)** 으로 출시합니다. 네이티브 앱(iOS/Android)은 출시 시점에 만들지 않으며, 후속 단계의 조건부 결정으로 미룹니다.

### 왜 PWA 우선인가

1. **유입 전체가 링크 공유·SEO 기반이다.**
   6모 “내 위치 보기” 챌린지, 카카오 오픈채팅 공유, 커뮤니티(수만휘·오르비) 배포, “2026 정시 환산점수”·“진학사 칸수 해석” 같은 검색 유입이 핵심 퍼널입니다. 이 퍼널의 생명은 *링크 탭 → 성적 입력 → 즉시 결과* 의 무마찰성입니다. 앱이면 그 사이에 스토어 이동·다운로드·설치·가입이 끼어들어 전환이 급감합니다. 검색 색인도 웹에서만 작동합니다.

2. **계절 상품이라 설치 동기가 약하다.**
   수요가 6모·9모·수능 직후로 집중되고 그 외 기간엔 거의 안 씁니다. 한두 번 쓰자고 앱을 설치·유지할 이유가 사용자에게 없습니다. PWA는 “쓸 때 들어오고 안 쓰면 잊는” 패턴에 정확히 맞고, 그러면서도 홈 화면 추가·웹푸시로 재방문 장치를 남깁니다.

3. **빠른 반복이 가능하다.**
   가격·UI·카피 A/B 테스트를 즉시 배포로 돌릴 수 있습니다. 앱은 매 변경이 재심사 대상이라, 시즌마다 데이터·환산식·문구가 바뀌는 이 제품에 치명적입니다.

4. **일정 리스크가 없다.**
   네이티브는 iOS 심사만으로 수일~수주가 걸리고, 입시·결제성 앱은 심사가 더 엄격합니다. 6모 선공개 일정에 묶이면 위험합니다.

5. **핵심 기능에 네이티브가 필요 없다.**
   성적 입력 폼, 환산점수 계산, 입결 차트, LLM 텍스트 리포트, PDF 생성, 공유 카드는 전부 웹 표준으로 구현됩니다. 성적표 OCR(후속)도 브라우저 카메라/파일 업로드로 가능합니다.

### 신뢰는 앱 여부가 아니라 데이터에서 나온다

“웹이라 신뢰가 떨어진다”는 우려는 이 시장에 해당하지 않습니다. 진학사·메가스터디·어디가 모두 웹 중심입니다. 신뢰는 데이터 원문 링크 제공(§7), 표본·변동성 명시(§2.4·§8.4), 면책 문구(§13.3)에서 만들어집니다.

### 네이티브 앱 전환 트리거 (후속, 조건부)

다음 조건이 충족될 때만 네이티브 앱을 검토합니다.

> **추합(추가합격) 실시간 알림이 리텐션·매출에 직접 기여한다는 것이 데이터로 확인될 때.**

정시 추합은 원서접수 마감·예비번호 변동·전화 추합 등 시간 민감 이벤트가 많아, 이 영역에서만 푸시 알림의 가치가 큽니다. 다만 이마저도 **1차로 웹푸시(PWA) + 카카오 알림톡으로 검증**한 뒤, ROI가 확인되면 네이티브로 전환합니다. 즉 네이티브는 “지울 카드”가 아니라 “2차로 미룬 카드”입니다.

### PWA 출시 시 명시적 비목표(Non-goals)

* 출시 시점 iOS/Android 네이티브 앱 (만들지 않음)
* 앱스토어/플레이스토어 등록 강제
* 설치 강제(install gate). 설치는 항상 선택적 권유

# 2. 제품 원칙

## 2.1 예측보다 해석

초기 제품은 진학사처럼 대규모 실지원 표본을 가진 서비스가 아닙니다.
따라서 “합격 확률”을 강하게 주장하면 안 됩니다.

제품의 첫 번째 원칙은 다음입니다.

> **확률을 단정하지 말고, 판단 근거를 설명한다.**

예시:

나쁜 표현:

> “이 대학은 합격 확률 78%입니다.”

좋은 표현:

> “전년도 입결 기준으로는 적정권에 가깝지만, 모집인원이 적고 실지원 표본 변동성이 커 교차검증이 필요합니다.”

## 2.2 시험별 단발 분석이 아니라 누적 타임라인

6모, 9모, 수능은 각각 별도 서비스가 아니라 하나의 타임라인입니다.

```text
나의 정시 타임라인

[6모] 현재 위치 진단
  ↓
[9모] 6모 대비 변화 분석
  ↓
[수능] 최종 원서 전략
  ↓
[원서접수] 가/나/다군 조합 저장
  ↓
[합불 발표] 실제 결과 인증
```

## 2.3 무료 계산기와 경쟁하지 않는다

환산점수 계산 자체는 무료 대체재가 많습니다.
Pacer의 유료 가치는 계산이 아니라 다음에 있습니다.

* 설명
* 교차검증
* 변화 추적
* 가/나/다군 조합
* 학부모용 리포트
* 컨설턴트 검수
* 실제 결과 기반 데이터 축적

## 2.4 불확실성을 숨기지 않는다

입시는 불확실합니다.
제품은 이 불확실성을 제거하는 척하면 안 됩니다.

반드시 표시해야 하는 불확실성 요소:

* 모집인원 변화
* 수시 이월
* 실지원자 표본
* 경쟁률
* 변환표준점수
* 대학별 환산식 변경
* 표본 수 부족
* 모집단위 선호도 변화
* 탐구 과목 유불리

## 2.5 수능 메인 이벤트를 기준으로 모든 구조를 설계한다

6모는 선공개입니다.
9모는 베타입니다.
수능이 메인 서비스입니다.

따라서 6모에서 반드시 확보해야 하는 것은 단순 분석 완료가 아니라 다음입니다.

* 계정
* 성적 데이터
* 목표 대학
* 재방문 알림
* 수능 시즌 전환 가능성
* 실제 결과 회수 동의

## 2.6 모바일 웹(PWA) 우선

모든 화면·플로우·성능 목표는 **모바일 브라우저를 1순위 환경**으로 두고 설계합니다.

* 설치 없이 링크로 즉시 가치를 경험하게 한다(무마찰 진입).
* 설치(홈 화면 추가)는 강제하지 않고, 가치 경험 후 자연스럽게 권유한다.
* 재방문 동력(9모·수능)은 단일 채널에 의존하지 않고 **카카오 알림톡·이메일·웹푸시 다중 채널**로 확보한다.
* 데스크톱은 반응형으로 지원하되, 컨설턴트용 다중 관리 화면(P2)만 데스크톱을 우선 고려한다.

# 3. 제품 범위

## 3.1 전체 제품 범위

Pacer는 다음 7개 모듈로 구성됩니다.

```text
1. 입시 사이클 대시보드
2. 성적 입력 및 변화 추적
3. 환산점수·입결 비교 엔진
4. AI 전략 리포트
5. 교차검증 모듈
6. 수능 최종 원서 조합 모듈
7. 실제 합불 결과 회수 모듈
```

이 7개 모듈은 모두 PWA(반응형 웹 + 서비스워커 + 웹푸시) 위에서 동작하며, 별도 네이티브 모듈은 포함하지 않습니다.

## 3.2 6모 선공개 범위

6모 버전의 목표는 **시장 공개, 유저 확보, 데이터 구조 검증**입니다.

포함 기능:

* 계정 생성 또는 익명 세션
* 6모 성적 입력
* 목표 대학/학과/지역 입력
* 현재 점수 구조 진단

P1:

* 휴대폰 번호 알림(알림톡)
* 보호자 계정 연결

### 필수 입력

* 사용자 유형: 학생 / 학부모 / 컨설턴트
* 입시연도

* 웹푸시 권한(보조 채널)
* 목표 대학
* 희망 지역
* 실제 결과 인증 참여 동의

### 알림 채널 우선순위 (재방문 설계의 핵심)

재방문(9모·수능)은 단일 채널에 의존하지 않습니다. 특히 **iOS Safari의 웹푸시는 ‘홈 화면에 추가(PWA 설치)’한 사용자에게만, iOS 16.4+에서만** 동작하므로, 미설치 사용자에게는 도달하지 않습니다. 따라서 채널 우선순위는 다음과 같습니다.

| 우선순위 | 채널 | 특징 | 대상 |
| --- | --- | --- | --- |
| 1 | 카카오 알림톡/친구톡 | OS 무관, 도달률 높음, 국내 친화 | 카카오 로그인/번호 동의자 |
| 2 | 이메일 | 비용 낮음, 보편적, 리포트 링크 첨부 용이 | 이메일 보유자 |
| 3 | 웹푸시(VAPID) | 무료·즉시성 높음, 단 iOS는 설치자 한정 | 웹푸시 허용 + (iOS는 홈화면 추가) 사용자 |

> 웹푸시 권한 요청은 가입 직후가 아니라, **9모 알림 신청 등 사용자가 알림 가치를 납득한 시점**에 요청합니다(권한 거부율 최소화).

처음부터 회원가입을 강제하면 이탈이 커집니다.
따라서 6모 선공개에서는 다음 구조가 적합합니다.

```text

→ 결과 일부 확인
→ 결과 저장하려면 가입
→ 9모 비교하려면 가입(+알림 채널 확보)
```

## 7.4 목표 설정 화면

### 목적

사용자의 분석 결과를 개인화하고, 이후 9모·수능 변화 추적의 기준점을 만듭니다.

### 입력 항목

* 희망 계열
* 대학 검색 선택 (`University.id`)
* 모집단위 검색 선택 (`AdmissionUnit.id`, 선택)
* 지원 성향
* 수시/정시 고민 정도

현재 구현 메모:

* 학생 UI는 DB의 active 2027 대학/모집단위를 검색형 combobox로 보여준다. 긴 dropdown이나 주관식 목표 문자열은 쓰지 않는다.
* 대학만 선택하면 해당 대학의 active 모집단위 전체를 분석 후보로 보고, 모집단위를 선택하면 선택한 모집단위를 직접 후보로 본다.
* 지역 데이터는 현재 대부분 `unknown`이라 v1 학생 화면에는 노출하지 않는다.
* 환산식·검수 원문·`formula_json`·`corrected_fields`는 admin/server 전용이며 학생 UI에서 수정하거나 내려받지 않는다.

### 지원 성향

| 값   | 설명          |
| --- | ----------- |
| 안정형 | 불합격 리스크 최소화 |
| 균형형 | 안정+적정+소신 균형 |
| 공격형 | 상향 지원 선호    |
| 미정  | 아직 판단 보류    |

### 수시/정시 고민 정도

| 값        | 설명        |
| -------- | --------- |
| 수시 중심    | 정시는 보조    |
| 수시·정시 병행 | 둘 다 고려    |
| 정시 중심    | 정시 가능성 중요 |
| 미정       | 판단 필요     |

### 저장 방식

목표 설정은 시험별로 snapshot을 남깁니다.

```text
6모 목표
9모 목표
수능 목표
```

왜냐하면 사용자의 목표 대학은 성적 변화에 따라 바뀌기 때문입니다.

# 8. 계산 엔진 명세

## 8.1 계산 엔진의 역할

LLM은 계산하지 않습니다.
계산 엔진이 판단 근거를 만들고, LLM은 그것을 설명합니다.

계산 엔진 역할:

1. 성적 검증
2. 성적 정규화
3. 지원 가능 조건 판정
4. 대학별 환산점수 계산
5. 전년도 입결 비교
6. 구간 분류
7. 신뢰도 산출
8. reason code 생성
9. 시험 간 변화 분석
10. 원서 조합 후보 생성

> 계산 엔진은 서버 사이드에서 수행합니다(PWA 클라이언트는 입력·표시만). 환산식/입결 데이터는 클라이언트로 원문 노출하지 않으며, 결과와 출처 링크만 전달합니다.

## 8.2 계산 방식

### 정확 환산

대학별 모집요강의 공식 환산식을 반영합니다.

적용 대상:

* 주요 상위권 대학
* 의치한약수
* 사용자가 많이 찾는 대학
* 수능 시즌 유료 기능 대상 대학

### 근사 비교

정확 환산식이 아직 구축되지 않은 모집단위에 사용합니다.

기준:

* 백분위 기반 비교
* 전년도 입결 기반
* 신뢰도 낮음 표시 필수

### 분석 불가

정보가 부족하거나 조건이 맞지 않으면 분석하지 않습니다.

표시 문구:

> 현재 데이터로는 이 모집단위의 정확한 분석을 제공하기 어렵습니다. 반드시 대학 입학처 모집요강을 함께 확인해 주세요.

## 8.3 구간 분류

기본 개념:

```text
score_gap = user_score - historical_cut_score
```

보정 요소:

* 모집인원 변화
* 충원율
* 소수 모집단위 여부
* 영어 감점 강도
* 탐구 변환표준점수 리스크
* 데이터 신뢰도
* 시험 시점, 6모/9모/수능

구간:

| 구간 | 설명            |
| -- | ------------- |
| 안정 | 전년도 기준 충분한 여유 |
| 적정 | 전년도 기준 근처     |
| 소신 | 일부 변수에 따라 가능  |
| 도전 | 점수 또는 변수 필요   |
| 위험 | 현재 기준 불리      |

## 8.4 신뢰도 라벨

각 분석 결과에는 신뢰도를 붙입니다.

| 신뢰도 | 조건                       |
| --- | ------------------------ |
| 높음  | 공식 환산식 검수 완료 + 입결 데이터 있음 |
| 중간  | 입결 있음 + 일부 근사 계산         |
| 낮음  | 근사 계산 중심                 |
| 제한  | 데이터 부족 또는 조건 불확실         |

신뢰도 설명 예시:

> 신뢰도: 중간
> 이유: 전년도 입결은 있으나, 해당 대학의 2027학년도 환산식 검수가 완료되지 않았습니다.

* 계산 결과 설명

## 13.2 데이터 저장 정책

* 비회원(익명 세션) 분석 결과는 제한 기간 후 삭제
* 회원은 직접 삭제 가능
* 성적 데이터는 암호화 또는 접근 제한

* 익명 통계 데이터와 개인 식별 정보 분리
* 웹푸시 구독 정보(endpoint/keys)는 별도 보관하고, 만료/해지 시 즉시 폐기

## 13.3 면책 문구

모든 결과 화면, 리포트, PDF 하단에 표시합니다.

> 본 서비스의 분석 결과는 공개 입시자료와 사용자가 입력한 성적을 바탕으로 한 참고용 정보입니다. 실제 합격 여부는 수능 본성적, 모집인원, 수시 이월, 지원자 표본, 경쟁률, 대학별 산출 방식 변경 등에 따라 달라질 수 있으며, 본 서비스는 합격을 보장하지 않습니다.

## 13.4 AI 사용 고지

> 일부 설명 리포트는 AI를 활용해 생성됩니다. AI는 입력된 성적과 계산 결과를 바탕으로 설명을 작성하며, 최종 지원 결정은 반드시 대학 입학처 모집요강과 전문가 상담을 함께 확인하시기 바랍니다.

## 13.5 미성년자·알림 동의

* 고3 사용자 상당수가 미성년자이므로, 알림 채널(특히 휴대폰 번호 기반 알림톡) 수집 시 연령·보호자 동의 안내를 명확히 한다.

### Service Worker

* 네트워크 우선(데이터 API) + 캐시 우선(정적 자산) 전략 분리
* 성적 입력 폼의 임시 상태 보존(오프라인/탭 전환 대비)
* push 이벤트 수신 → notification 표시 / notificationclick → 해당 리포트로 딥링크

### 설치(홈 화면 추가) 권유

* `beforeinstallprompt` 캡처 후, **첫 결과 확인 이후**에만 커스텀 권유 노출
* iOS Safari는 자동 프롬프트 미지원 → “공유 → 홈 화면에 추가” 안내 카드 제공
* 설치 강제 금지(항상 선택)

### 웹푸시(Web Push)

* VAPID 키로 PushSubscription 생성, NotificationSubscription에 저장
* **iOS 제약 명시**: iOS 16.4+ & 홈 화면에 추가(PWA 설치)한 경우에만 웹푸시 수신 가능. 미설치 iOS 사용자는 웹푸시 도달 불가 → 알림톡/이메일로 커버
* 발송 이벤트: 9모 오픈, 수능 성적표 공개, (검증 시) 추합/원서마감 등 시간 민감 알림
* 만료/실패 endpoint는 배치로 정리

## 17.5 알림(재방문) 아키텍처

재방문은 제품의 척추이므로 단일 채널 의존을 피합니다.

```text
재방문 트리거(이벤트): september_mock_open / csat_score_open / waitlist_update
  ↓
Notification Service
  ├─ 1순위: 카카오 알림톡  (OS 무관, 도달률 높음)
  ├─ 2순위: 이메일        (보편적, 리포트 링크 첨부)
  └─ 3순위: 웹푸시        (즉시성, iOS는 설치자 한정)
  ↓
딥링크 → 해당 시험 분석/리포트 화면
```

* 사용자별로 보유 채널 중 우선순위에 따라 발송(중복 최소화)
* 채널별 도달·클릭·전환을 로깅(§16.5 reminder_opt_in / return_from_reminder에 channel 파라미터)

## 17.6 성능·호환성 목표

* 주요 모바일 브라우저(안드로이드 Chrome, iOS Safari) 우선 지원
* Lighthouse PWA 체크 통과(설치 가능, 오프라인 셸, 매니페스트 유효)
* 모바일 성적 입력 3분 내, 분석 결과 표시 10초 내(§18.3)
* 수능 시즌 트래픽 급증 대비: 기본 분석 우선 제공 + LLM 리포트 비동기 생성(§18.4)

# 23. 최종 요약

이 제품은 단순한 “6모 분석기”가 아니라, **수능 최종 원서 전략을 메인으로 하는 정시 시즌 대시보드**이며, **앱스토어 없이 링크로 진입하는 모바일 웹앱(PWA)** 으로 전달합니다.

6모에서는 현재 위치를 보여주고, 사용자를 저장하고 알림 채널을 확보합니다.
9모에서는 변화 추적을 보여주고, 재방문과 유료화를 검증합니다.
수능에서는 교차검증과 가/나/다군 조합을 제공하며, 본격적으로 매출을 만듭니다.
수능 이후에는 실제 합불 데이터를 회수해 다음 해의 데이터 해자를 만듭니다.

따라서 최종 제품의 중심 구조는 다음입니다.

```text
User
  ↓
AdmissionCycle
  ↓
ExamScore: 6모 / 9모 / 수능
  ↓
AnalysisSnapshot
  ↓
StrategyReport
  ↓
SavedAdmissionUnit
  ↓
ApplicationPlan
  ↓
FinalOutcome
```

가장 중요한 P0 판단은 두 가지입니다.

> **(1) 6모 선공개 버전에서도 반드시 AdmissionCycle, 결과 저장, 목표 대학 저장, 9모·수능 재방문 장치를 넣어야 한다.**
> **(2) 전달은 PWA로 시작하고, 재방문 알림은 카카오 알림톡·이메일·웹푸시 다중 채널로 확보한다. 네이티브 앱은 추합 실시간 알림 ROI가 검증된 뒤에만 검토한다.**

이 구조로 만들면 6모에서 만든 제품이 버려지지 않고, 그대로 9모와 수능 메인 서비스로 확장되며, 플랫폼(PWA) 또한 동일 코드베이스로 시즌 내내 빠르게 반복·확장됩니다.
